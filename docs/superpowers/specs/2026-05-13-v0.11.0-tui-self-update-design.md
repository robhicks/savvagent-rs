# TUI Self Update — v0.11.0 design

Date: 2026-05-13
Status: pending review
Roadmap: v0.11.0 (#2)
Supersedes: nothing
Related: v0.10.0 localize-tui design (locales infra used by the banner i18n)

## Problem

A user running an installed `savvagent` binary has no in-band signal when a
newer release is available. Pre-1.0 ships frequently and the wire/protocol
surfaces are still moving (see `crates/savvagent-protocol/SPEC.md`); users on
stale binaries miss provider/tool fixes and can hit subtle protocol
mismatches against newer in-process components. Issue #2 specifies an
asynchronous check on launch and a `/update` slash command that performs
the upgrade.

## Approach

A single new built-in plugin, `internal:self-update`, owns the entire
feature. The plugin:

1. Subscribes to `HookKind::HostStarting`. On `HostEvent::HostStarting`,
   spawns an async task that performs the version check (network), respects
   the 24h cache, and stashes the latest-version result in plugin state.
2. Contributes to a new render slot `home.banner`, painted above the
   existing `home.tips` line. When the cached result indicates a newer
   release, the slot renders a one-line banner; otherwise it renders
   nothing.
3. Registers a `/update` slash command. When invoked, it consults the same
   cached result and shells the [`self_update`](https://crates.io/crates/self_update)
   crate to fetch the matching cargo-dist tarball asset for the current
   target triple, verify, extract, and atomically swap the running binary.
   On success it transitions the banner text from "update available" to
   "updated to vX, restart to apply" and emits a one-line stdout hint on
   quit.

The cargo-dist tarball is the only real install path: cargo-dist's `shell`
and `powershell` installers drop the binary in `$CARGO_HOME/bin`, which is
also where `cargo install` lands — so the issue's three-way install-method
split collapses to two cases the plugin distinguishes: `Installed` (any
binary outside the workspace `target/` tree) and `Dev` (any binary whose
parent directory is `target/{debug,release}/` under a Cargo workspace).
Dev builds skip the network check entirely and `/update` prints a hint.

Opt-out lives at the CLI/environment layer for v0.11.0:
`SAVVAGENT_NO_UPDATE_CHECK=1` and `--no-update-check`. The issue's third
opt-out vector (`~/.savvagent/config.toml`) is deferred — savvagent has no
config.toml today and introducing one for a single boolean is premature.
A follow-up issue will track adding it when there's a second knob to
justify the file.

## Modules

```
crates/savvagent/src/plugin/builtin/self_update/
├── mod.rs            // SelfUpdatePlugin + manifest + slot/slash dispatch
├── install_method.rs // InstallMethod enum + detect() (pure, unit-testable)
├── check.rs          // GitHub Releases query + semver compare + cache file
└── apply.rs          // /update slash handler → self_update integration
```

The plugin registers in `crates/savvagent/src/plugin/builtin/mod.rs` and
appends to the `register_builtins()` vector in
`crates/savvagent/src/plugin/mod.rs`.

UI wiring lives in `crates/savvagent/src/ui.rs`: the vertical layout
constraints gain a `Constraint::Length(1)` row for the banner between the
log and the tips line, and `compute_home_frame_data` adds a fourth slot
render for `home.banner`. The banner row is unconditionally one line tall;
when the plugin contributes nothing, the row paints background only.

## Data flow

```
HostEvent::HostStarting
    │
    ▼
SelfUpdatePlugin::on_event ──spawn──▶ tokio::task (check.rs)
                                          │
                                          ├──▶ read ~/.savvagent/update-check.json (cache)
                                          │     if cached_at + 24h > now: use cached
                                          │     else: GET github API
                                          │
                                          ▼
                                     stash UpdateState into plugin (Mutex<Option<...>>)

render_slot("home.banner", …)
    │
    └──▶ read plugin state → emit StyledLine (or nothing)

handle_slash("update", …)
    │
    └──▶ read plugin state; require Available; call apply.rs (self_update)
         → on success, mutate state to Updated { restart_to_apply: vX }
```

State machine values:

```
enum UpdateState {
    Unknown,           // initial state, before HostStarting completes
    Disabled,          // dev build, or opt-out
    UpToDate,
    Available { current: Version, latest: Version, asset_url: String },
    Updated { from: Version, to: Version },
    CheckFailed,       // logged at debug; UI shows nothing
}
```

## Error handling

Every network/parse/permission/IO error path returns to `UpdateState::CheckFailed`
or `CheckFailed`-equivalent silent failure. Failures log at
`tracing::debug!` only. The UI never surfaces a check failure — that would
contradict the issue's "fail silent" requirement.

`/update` failures are the exception: a user-initiated command should
surface its outcome. On failure the plugin keeps the banner in `Available`
state and emits an in-conversation note via `app.push_note(...)`-equivalent
plugin effect (`Effect::PushNote`).

## Testing

- `install_method.rs`: pure unit tests over a `detect_from_path(&Path)`
  helper that takes the exe path as input. The plugin's runtime entry
  point wraps it around `std::env::current_exe()`.
- `check.rs`: unit tests over a `compare_versions(current: &str, latest: &str)`
  pure helper covering equal, ahead, behind, and pre-release tags. The
  HTTP layer is tested separately with a small mocked reqwest middleware
  (or by injecting a callable trait so the test substitutes a stub).
- `apply.rs`: not unit-tested for the binary swap itself — `self_update`
  has its own coverage. We assert only that the plugin transitions state
  correctly when the swap returns Ok / Err.
- Integration: an end-to-end smoke test that constructs the plugin,
  feeds it `HostEvent::HostStarting` with `SAVVAGENT_NO_UPDATE_CHECK=1`,
  and asserts the resulting state is `Disabled` with no network calls
  made.

## i18n keys

Three new keys per locale (en/es/pt/hi):

- `self-update.banner-available` — "Update available: {current} → {latest}  (run /update)"
- `self-update.banner-updated` — "Updated to v{version}. Restart savvagent to apply."
- `self-update.slash-summary` — "Apply the latest release"

Parity test (added in v0.10.0 PR 5) enforces that all four locales define
the same key set.

## PR slicing

1. **PR 1 — scaffold.** New plugin module with manifest only; `InstallMethod`
   enum + `detect_from_path` helper + unit tests. Registered in
   `register_builtins`. No network, no slot wiring, no `/update`. This PR
   also commits the design spec.
2. **PR 2 — version check.** `check.rs`: reqwest GitHub Releases query,
   semver compare, the `UpdateState` enum, the `HostStarting` hook
   wiring. Tests with a stubbed HTTP fetcher. Still no UI surface.
3. **PR 3 — banner UX.** New `home.banner` slot wired into `ui.rs`,
   plugin's `render_slot` impl, i18n keys for en/es/pt/hi, opt-out flags
   (env var + CLI). End-to-end visible behavior.
4. **PR 4 — `/update` apply.** `apply.rs` with `self_update` integration;
   slash command handler; restart-hint state transition and on-quit
   stdout hint.
5. **PR 5 — cache + ship.** `~/.savvagent/update-check.json` 24h cache.
   Workspace version bump to `0.11.0` (and matching `[workspace.dependencies]`
   literal bumps per memory). Release notes / GitHub Release body.
   Close issue #2.

## Out of scope (per issue)

- Auto-apply without confirmation
- Channel/track selection (stable vs nightly)
- Signing/notarization of release artifacts
- `~/.savvagent/config.toml` introduction (deferred to its own issue)
