## Problem

`savvagent`'s TUI and host have outgrown the structure that fit v0.1–v0.8.
The pain shows up at the seams:

- **Modals don't compose.** Eight `InputMode` variants live as flat fields
  on `App`, with key-handler arms in `main.rs` and render branches in
  `ui.rs`. The v0.8 theme picker had to thread three new fields through
  this state machine because each modal is a one-off. Adding a ninth
  modal (`/plugins`, file viewer, file editor) is straight copy-paste —
  the structure does not generalize.
- **Slash commands are statically wired.** `palette.rs` enumerates the
  built-in command list at compile time. Adding `/edit` means editing
  that file *and* `main.rs`'s dispatcher *and* `ui.rs`'s palette render.
  There is no place to ask "what slash commands does this build offer?"
  except by reading code.
- **The footer is a single rendered line.** It currently bakes in
  provider+model+sandbox+turn-state+working-dir as one
  `Block::default()` call in `ui.rs`. Splitting it for any reason
  (Anthropic-specific badge, sandbox state visibility, future MCP server
  count) means re-laying-out by hand and adding more conditionals.
- **Providers and tools are MCP-shaped externally but ad-hoc internally.**
  The TUI hard-wires Anthropic/OpenAI/Gemini/Local in `providers.rs`. A
  contributor adding a fifth provider must touch the TUI again — the
  point of the MCP-everywhere bet was to push that out of the host.

The shared root cause: there's no abstraction for "a unit of TUI/host
behavior that can be enumerated, enabled/disabled, contributed by code
outside the host crate, and (eventually) loaded from WASM." Every
existing extension story is bespoke.

v0.9.0 introduces that abstraction.

## Approach

A new leaf crate `savvagent-plugin` defines two traits — `Plugin` and
`Screen` — and the data types that cross plugin boundaries. The
`savvagent` crate gains a small runtime that constructs 17 built-in
plugin instances at startup, indexes their contributions across seven
contribution kinds (slash commands, screens, themes, providers, hooks,
render slots, keybindings), and dispatches host events / key events /
slash commands through those indexes. The existing `InputMode` state
machine is replaced by a screen stack. The footer becomes three
slot-routed segments and a new tips line appears above the prompt.

Every plugin's value comes back to the runtime as `Vec<Effect>` — a
closed enum of typed operations (push a note, open a screen, set the
active theme, etc.). `apply_effects(&mut App, Vec<Effect>)` is the
**only** place `App` mutates in response to plugin output. This is the
same boundary shape WIT host functions will need when this becomes
WASM, deferred but binding (§9).

Trait surface, types, and effect vocabulary are designed under WIT
portability rules from day one (§9). No `.wit` file ships in v0.9.0;
the promise is only that the surface, if frozen, could be mechanically
translated to WIT.

The v0.8 theme picker is the case study for how all 17 extractions go
(§7). It exercises per-open state, live preview, cancel-reverts,
screen-owned tips, and the slash+screen+catalog bundling.

## Architecture

### New crate: `savvagent-plugin`

A leaf crate with zero runtime behavior:

```
crates/savvagent-plugin/
    Cargo.toml          # async-trait only; no ratatui/crossterm/tokio/anyhow
    src/
        lib.rs          # re-exports
        plugin.rs       # Plugin trait
        screen.rs       # Screen trait
        manifest.rs     # Manifest, Contributions, *Spec types
        effect.rs       # Effect enum
        event.rs        # HostEvent enum + HookKind enum
        styled.rs       # StyledLine, StyledSpan, ThemeColor, TextMods
        types.rs        # Region, KeyEventPortable, KeyCodePortable, KeyMods,
                        # ScreenArgs, PluginId, ProviderId, Timestamp,
                        # ScreenInstanceId, PluginKind, ScreenLayout, KeyScope,
                        # BoundAction, ThemeEntry, ThemePalette, PluginError
```

Workspace `[workspace.dependencies]` gains:

```toml
savvagent-plugin = { path = "crates/savvagent-plugin", version = "0.9.0" }
```

Consumers in v0.9.0:
- `savvagent` (TUI/runtime) — dispatches plugin trait calls.
- `savvagent-host` — emits `HostEvent`s; reads typed effect payloads
  from `Effect::RegisterProvider` etc.

`provider-*` and `tool-*` crates do **not** depend on `savvagent-plugin`
in v0.9.0. The "providers as plugins" pattern means each provider has a
thin shim plugin (`internal:provider-anthropic`) living *inside* the
`savvagent` crate's `plugin/builtin/` tree; the provider crates
themselves keep their existing surface.

### Trait surface

```rust
// crates/savvagent-plugin/src/plugin.rs
use crate::{Effect, HostEvent, Manifest, PluginError, Region, ScreenArgs,
            StyledLine, ThemeEntry};

#[async_trait::async_trait]
pub trait Plugin: Send + Sync {
    fn manifest(&self) -> Manifest;

    /// Dispatch for a slash command this plugin declared. `name` is the
    /// bare command without the leading '/'.
    async fn handle_slash(
        &mut self,
        name: &str,
        args: Vec<String>,
    ) -> Result<Vec<Effect>, PluginError> {
        let _ = (name, args);
        Ok(vec![])
    }

    /// Factory called when `Effect::OpenScreen { id, .. }` targets a
    /// screen declared in this plugin's manifest. Each open creates a
    /// fresh `Screen` instance (per-open state, see §3).
    fn create_screen(
        &self,
        id: &str,
        args: ScreenArgs,
    ) -> Result<Box<dyn Screen>, PluginError> {
        let _ = args;
        Err(PluginError::ScreenNotFound(id.to_string()))
    }

    /// Awaited event hook. Plugin returns effects; host applies them
    /// before continuing the event loop.
    async fn on_event(
        &mut self,
        event: HostEvent,
    ) -> Result<Vec<Effect>, PluginError> {
        let _ = event;
        Ok(vec![])
    }

    /// Render a slot this plugin declared in its manifest. Called every
    /// frame; should be cheap and side-effect-free.
    fn render_slot(&self, slot_id: &str, region: Region) -> Vec<StyledLine> {
        let _ = (slot_id, region);
        vec![]
    }

    /// Static theme catalog. Pulled once at registration time.
    fn themes(&self) -> Vec<ThemeEntry> {
        vec![]
    }
}

// crates/savvagent-plugin/src/screen.rs
use crate::{Effect, HostEvent, KeyEventPortable, PluginError, Region,
            StyledLine};

#[async_trait::async_trait]
pub trait Screen: Send {
    fn id(&self) -> &str;

    fn render(&self, region: Region) -> Vec<StyledLine>;

    async fn on_key(
        &mut self,
        key: KeyEventPortable,
    ) -> Result<Vec<Effect>, PluginError>;

    async fn on_event(
        &mut self,
        event: HostEvent,
    ) -> Result<Vec<Effect>, PluginError> {
        let _ = event;
        Ok(vec![])
    }

    /// Optional tips line for the active screen. When a screen is on top
    /// of the stack, the host renders these tips instead of polling the
    /// `home.tips` slot. Default: no tips.
    fn tips(&self) -> Vec<StyledLine> {
        vec![]
    }
}
```

## Contribution kinds

### 1. Slash commands

`SlashSpec { name, summary, args_hint }`. Dispatched by `SlashRouter`.

- **Conflict resolution:** hard error at startup. If two enabled plugins
  both register a `name`, `register_builtins()` returns an error and the
  TUI fails to start with a message identifying both plugins. User
  recourse is `plugins.toml` (disable one) or filing a bug.
- The reserved namespace `internal:` may appear in plugin IDs but not in
  slash names. Slash names are flat: `theme`, not `internal:themes:theme`.
- `/theme list` is **one slash with arg `["list"]`**, not two slashes. The
  palette shows two completions sharing one underlying SlashSpec where
  arg variants are commonly used; v0.9 keeps this simple by listing the
  base name only and letting the palette show `args_hint`.

### 2. Screens

`ScreenSpec { id, layout: ScreenLayout }`.

- Layouts: `Fullscreen`, `CenteredModal`, `BottomSheet`. Closed enum —
  no plugin-defined layouts in v0.9.0.
- The host owns chrome (border, title, centering, dim-background). The
  plugin only fills the inner content region via `Screen::render`. This
  unifies theme picker / palette / splash / view-file under the same
  primitives.
- One Plugin can declare multiple Screens (e.g., `internal:connect`
  declares only `connect.picker`; `internal:themes` declares only
  `themes.picker`; a hypothetical future provider plugin could declare
  both `model.picker` and `auth.picker`).
- **Screen ID conflicts:** hard error at startup. Two enabled plugins
  cannot declare the same `ScreenSpec::id`.

### 3. Themes

`ThemeEntry { slug, label, dark, palette }`.

- Registry concatenates contributions from all enabled plugins into a
  single catalog keyed by slug.
- **Slug conflicts:** later-registered wins with a startup warning to
  `tracing::warn!`. Themes are cosmetic; this is the right place for
  forgiveness so future user themes don't accidentally brick the TUI.
- The active theme slug is owned by `App` and persisted via the
  existing config file (not `plugins.toml`). `Effect::SetActiveTheme {
  slug, persist }` is the only way to mutate it.

### 4. Providers

`ProviderSpec { id, display_name, requires_credential, in_process }`.

- The plugin is a thin shim that owns the `ProviderClient` lifecycle.
  Construction happens on `Hook(HostStarting)` for credentialed
  providers (read keyring → register) or on demand from
  `handle_slash("connect", …)` (user-triggered auth).
- The user-facing surface (slash command + manager visibility) is
  expressed through the WIT-portable `Plugin` trait — the plugin emits
  `Effect::RegisterProvider { id, display_name }` to *announce* the
  provider, which the runtime adds to its connect-picker list.
- **The actual `Box<dyn ProviderClient>` hand-off is a savvagent-internal
  seam, not a `savvagent-plugin` API.** A separate Rust trait
  `BuiltinProviderPlugin` lives in `crates/savvagent/src/plugin/builtin/
  provider_common.rs` (alongside the four provider plugin
  implementations); it has methods like
  `take_client(&mut self) -> Option<Box<dyn ProviderClient>>` that the
  runtime calls after `RegisterProvider` is observed. This explicitly
  steps outside the WIT-portable surface because `dyn ProviderClient`
  is not WIT-portable (rule §9.2). v1.0+ will replace this seam with a
  WIT-resourced port; v0.9.0 acknowledges the limitation in the spec
  rather than papering over it.
- A provider plugin disabled in `plugins.toml` simply does not register —
  the provider disappears from the connect picker.

### 5. Hooks

`HookKind` is the closed enum listed above. `HostEvent` is its sibling
payload type.

- **Dispatch:** sequential, awaited. For each event, the dispatcher
  iterates subscribers in registration order, awaits each
  `plugin.on_event(event.clone())`, and accumulates returned effects.
  Effects are applied after all subscribers have run.
- **Why sequential:** simplest mental model, deterministic ordering,
  matches Approach A's discipline of low integration risk per PR. If a
  hook becomes a bottleneck post-v0.9, the dispatcher can be relaxed to
  parallel without changing trait shape.
- **Streaming-delta hooks are explicitly out of scope.** No
  `on_token`, `on_chunk`, `on_partial_message`. The hook frequency cap
  is once-per-turn-event.

### 6. Render slots

`SlotSpec { slot_id, priority }`.

- v0.9.0 slot IDs: `home.footer.left`, `home.footer.center`,
  `home.footer.right`, `home.tips`. Closed set in v0.9.0; the
  abstraction is general for future `home.sidebar.*` etc.
- Multiple plugins may target the same slot. Host orders by `priority`
  (ascending) and renders each contributor's `Vec<StyledLine>` in turn.
- Footer slots: horizontal concatenation within the slot's third of the
  terminal; left/center/right-aligned within their segment; truncated
  with `…` if over-wide.
- Tips slot: a single row; spans concatenated left-to-right with
  single-space separators between contributors.
- Built-in priority budget: `0..=999`. Third-party plugins (v1.0+)
  reserve `1000..=i32::MAX`.

### 7. Keybindings

`KeybindingSpec { chord, scope, action }`.

- `KeyScope`: `Global` | `OnHome` | `OnScreen(screen_id)`.
- `BoundAction`: `RunSlash { name, args }` | `EmitEffect(Effect)`.
- **Conflict resolution within a scope:** hard error at startup.
- **Reserved chords (not pluggable):** `Ctrl-C` (cancel active operation),
  `Ctrl-D` (quit). These stay hard-coded in the runtime so a buggy
  plugin can't trap the user.

### Registry & manifests

```rust
pub struct PluginRegistry {
    plugins: HashMap<PluginId, Arc<Mutex<Box<dyn Plugin>>>>,
    enabled: HashSet<PluginId>,
    indexes: Indexes,
}

struct Indexes {
    slash:       HashMap<String, PluginId>,
    slots:       HashMap<String, Vec<(i32, PluginId)>>,   // sorted by priority
    hooks:       HashMap<HookKind, Vec<PluginId>>,
    keybindings: HashMap<(KeyScope, ChordPortable), (PluginId, BoundAction)>,
    screens:     HashMap<String, PluginId>,
}
```

Startup order:
1. `register_builtins()` constructs all 17 plugins.
2. Read `plugins.toml`; apply enabled/disabled to `enabled` set. Core
   plugins are forced enabled regardless of file content.
3. Build `Indexes` from each enabled plugin's `manifest()`.
4. Emit `HostEvent::HostStarting` through `HookDispatcher::emit`.
5. Drain returned effects through `apply_effects`.

Toggle handling (manager screen):
1. Update `enabled` set.
2. Rebuild `Indexes` from scratch (cheap — 17 plugins).
3. Atomically write `plugins.toml`.
4. No `HostEvent` is emitted for toggles in v0.9; future hooks like
   `PluginEnabled`/`PluginDisabled` are deferred.

### Screen stack

```rust
pub struct ScreenStack {
    stack: Vec<Box<dyn Screen>>,
    next_instance_id: u32,
}

impl ScreenStack {
    pub fn top(&self) -> Option<&dyn Screen> { /* ... */ }
    pub fn top_mut(&mut self) -> Option<&mut dyn Screen> { /* ... */ }
    pub fn push(&mut self, screen: Box<dyn Screen>);
    pub fn pop(&mut self) -> Option<Box<dyn Screen>>;
    pub fn depth(&self) -> usize { self.stack.len() }
}
```

- Replaces every variant of the existing `InputMode` enum. No flat
  fields on `App` remain to model "is a modal active."
- Key events route to `top_mut()` if present; otherwise through
  `KeybindingRouter` (OnHome scope first, then Global).
- `Effect::OpenScreen { id, args }` resolves `id` → `PluginId` via
  `Indexes::screens`, calls `plugin.create_screen(id, args)`, and pushes
  the returned `Box<dyn Screen>`.
- `Effect::CloseScreen` pops. Popping an empty stack is a no-op (logged
  at debug).

```
for region in [home.footer.left, home.footer.center, home.footer.right] {
```

### Hook dispatcher

```rust
pub struct HookDispatcher<'r> { indexes: &'r Indexes }

impl<'r> HookDispatcher<'r> {
    pub async fn emit(&self, event: HostEvent) -> Vec<Effect> {
        let kind = event.kind();
        let mut all_effects = vec![];
        for plugin_id in &self.indexes.hooks.get(&kind).unwrap_or(&vec![]) {
            let plugin = self.registry.get(plugin_id);
            match plugin.lock().await.on_event(event.clone()).await {
                Ok(effs) => all_effects.extend(effs),
                Err(e) => tracing::warn!(plugin = %plugin_id, error = ?e,
                                         "plugin hook errored"),
            }
        }
        all_effects
    }
}
```

### Effects applier

`apply_effects(app: &mut App, effects: Vec<Effect>)` is the **only**
mutation surface for `App` in response to plugin output. Each effect
maps to a small `App` method:

| Effect variant | App side effect |
|----------------|----------------|
| `PushNote { line }` | `app.push_styled_note(line)` |
| `OpenScreen { id, args }` | `app.screen_stack.push(...)` |
| `CloseScreen` | `app.screen_stack.pop()` |
| `SetActiveTheme { slug, persist }` | `app.set_active_theme(slug); if persist { app.persist_config() }` |
| `SetActiveProvider { id, persist }` | `app.set_active_provider(id); if persist { app.persist_config() }` |
| `RegisterProvider { id, display_name }` | `app.providers.register(id, display_name)` |
| `SaveTranscript { path }` | `app.save_transcript_to(path)` |
| `PromptSend { text }` | `app.submit_prompt(text)` |
| `RunSlash { name, args }` | re-enters `slash_router.dispatch(name, args)` (re-entrancy is bounded: depth > 4 is a hard error to prevent infinite loops) |
| `ClearLog` | `app.clear_log()` |
| `Quit` | `app.request_quit()` |
| `Stack(children)` | recurse over children in order |

### Event loop (post-v0.9)

```
1. Block on next event (terminal input | host channel | tick).
2. If terminal key:
     if let Some(screen) = stack.top_mut():
         effects = screen.on_key(key).await?
     else:
         (plugin, action) = keybindings.route(key, OnHome → Global)?
         effects = action.execute()
   else if host event:
     effects = hook_dispatcher.emit(event).await
3. apply_effects(&mut app, effects)
4. Render:
     home regions (log + tips + prompt + footer segments via slot router)
     if let Some(screen) = stack.top():
         paint_chrome(screen.layout())
         paint_lines(screen.render(inner_region))
         paint_tips(screen.tips())
5. Loop.
```

## Multi-region TUI layout

```
┌────────────────────────────────────────────────────────────┐
│ conversation log (existing render_log)                     │
│                                                            │ ← Fill remaining
│                                                            │
├────────────────────────────────────────────────────────────┤
│ tips line: home.tips slot (or active screen's tips())      │ ← 1 row
├────────────────────────────────────────────────────────────┤
│ prompt input (existing tui-textarea)                       │ ← Auto height
├────────────────────────────────────────────────────────────┤
│ [footer.left]    [footer.center]    [footer.right]         │ ← 1 row
└────────────────────────────────────────────────────────────┘
```

Notes:
- Each footer segment receives a third of the terminal width. Within
  each segment, contributions render left-aligned (left), centered
  (center), or right-aligned (right).
- When a screen is on top of the stack, it occupies the central region
  (everything above the tips line) according to its `ScreenLayout`. The
  tips and footer remain visible unless the screen's layout is
  `Fullscreen { hide_chrome: true }`.
- Tips line: if the active screen returns non-empty `tips()`, those are
  rendered. Otherwise the host queries the `home.tips` slot.

Footer/tips built-in contributors (v0.9.0):

| Slot | Contributing plugin | Content |
|------|--------------------|---------|
| `home.footer.left`   | `internal:provider-*` (whichever is active) | active provider name + model |
| `home.footer.center` | `internal:home-footer`                      | sandbox state + turn state |
| `home.footer.right`  | `internal:home-footer`                      | working dir, key reminder `? for help` |
| `home.tips`          | `internal:home-tips`                        | static "Press / for commands" |
| `home.tips`          | `internal:command-palette`                  | palette-availability hint |
| `home.tips`          | `internal:splash`                           | connect-progress hint during warm-up |

## Persistence

`~/.savvagent/plugins.toml`:

```toml
# schema_version determines forward compat
schema_version = 1

# Optional plugins only. Core plugins are never written here.
# Plugins not listed default to enabled.
[plugins."internal:provider-anthropic"]
enabled = true

[plugins."internal:provider-local"]
enabled = false

[plugins."internal:view-file"]
enabled = true
```

Robustness rules:
- File missing → empty file (all optional plugins default enabled).
- Unknown plugin id in file → skip with `tracing::warn!`; do not error.
- `schema_version` mismatch → log warning; ignore the file entirely
  (all optional plugins default enabled). Do not partial-parse.
- Core plugin appearing in the file → silently ignored.
- Manager writes via `tempfile + rename` for atomicity.
- Directory `~/.savvagent/` is created with `0o700` if missing; file
  with `0o600`. (Same conventions as the existing
  `~/.savvagent/credentials` keyring fallback.)

## WIT portability rules

Deferred but binding. Every type and method in `savvagent-plugin`
follows these rules so the v1.0 WIT port is mechanical.

**Type-system rules.**

1. **No borrows in trait signatures.** All inputs and outputs are owned
   (`String`, `Vec<T>`, owned structs/enums). `&str` is tolerated only
   when the value is consumed and not stored.
2. **No `Arc<dyn Trait>` or `Rc<…>` in public types.** Resources cross
   as opaque IDs (`PluginId`, `ScreenInstanceId`) or via factory methods
   returning `Box<dyn Trait>` (`create_screen`).
3. **Errors are concrete enums.** `PluginError` is a closed
   `#[non_exhaustive]` enum. No `anyhow::Error`, no `Box<dyn Error>`.
4. **No callbacks, no closures in trait sigs.** Plugins return
   `Vec<Effect>`; they do not hold a host callback handle. The
   `apply_effects`-only-mutation-surface (§3) is what makes this work.
5. **Explicit-width numerics.** `u16` for terminal dimensions, `u32`
   for indices and turn IDs, `i32` for priorities, `i64+u32` for
   timestamps. No `usize`/`isize` in public types.
6. **No `serde_json::Value`** in trait sigs. Even with typed effects in
   force, this is the belt-and-suspenders rule for future contributors.
7. **Async restricted to `async_trait`** on `Plugin` and `Screen`. No
   futures-in-types, no `Stream` returns, no `tokio::sync::*` handles.

**Convenient Rust → WIT-portable replacements.**

| Convenient Rust | WIT-portable replacement |
|-----------------|--------------------------|
| `crossterm::event::KeyEvent` | `KeyEventPortable { code, modifiers }` |
| `ratatui::layout::Rect` | `Region { x, y, width, height }` (all `u16`) |
| `ratatui::style::Color` | `ThemeColor` enum (16 named + Indexed(u8) + Rgb{r,g,b}) |
| `ratatui::style::Modifier` (bitflags) | `TextMods { bold, italic, underline, reverse, dim }` |
| `std::path::PathBuf` | `String` (UTF-8 path; rejected at boundary if non-UTF8) |
| `std::time::SystemTime` | `Timestamp { secs: i64, nanos: u32 }` |
| `tokio::sync::oneshot::Sender<T>` | not exposed — use `Effect` round-trip |
| `bitflags::bitflags!` flags | structs of `bool` fields |

These mappings live in `savvagent-plugin` as the canonical types.
`From`/`Into` impls between portable and ratatui/crossterm types live
in the runtime (`crates/savvagent/src/plugin/convert.rs`), not in the
leaf crate.

**Crate-level guard.** `savvagent-plugin/Cargo.toml` omits `ratatui`,
`crossterm`, `tokio`, `anyhow` from `[dependencies]`. CI lints this
section. A PR that adds any of those four deps without a
`# wit-portable: justified` comment fails the dep-guard check.

Implementation: a one-line `grep -E` check in CI is sufficient; no
clippy lint required.

**What this section is *not*.** Not a commitment to do the WIT port in
v1.0 or in any timeframe. v0.9.0 ships no `.wit` file. The promise is
only that the v0.9 trait surface, if frozen, could be mechanically
translated.

## Theme picker migration (case study)

The v0.8 theme picker is the test case for how every extraction works.
It exercises all five hard shapes any extraction will hit:

1. Per-open state that doesn't survive close (filter, cursor).
2. Live preview that mutates `App` while the screen is open.
3. Cancel-reverts semantics.
4. Screen-owned tips line.
5. Slash command + Screen + Theme catalog bundled in one plugin.

### PR 5.5 (pre-extraction refactor)

A functionally no-op refactor lifts picker state from `App` into a
self-contained struct. This lands as its own small PR ahead of PR 6.

```rust
// crates/savvagent/src/theme/picker.rs (new)
pub struct ThemePicker {
    filter: String,
    cursor: usize,
    pre_open_theme: ThemeSlug,
}

impl ThemePicker {
    pub fn new(active: ThemeSlug, _catalog: &ThemeCatalog) -> Self { /* ... */ }
    pub fn on_key(&mut self, key: KeyEvent) -> PickerOutcome { /* ... */ }
    pub fn render(&self, area: Rect, buf: &mut Buffer, catalog: &ThemeCatalog) { /* ... */ }
}

pub enum PickerOutcome {
    Stay,
    PreviewTheme(ThemeSlug),
    Apply(ThemeSlug),
    Cancel,
}
```

`App` now holds `Option<ThemePicker>` instead of three flat fields.
`main.rs` routes keys to it when present; `ui.rs` calls
`picker.render(...)` when present. All v0.8 picker tests pass without
modification.

### PR 6 (extraction)

```
crates/savvagent/src/plugin/builtin/themes/
    mod.rs           // ThemesPlugin impl Plugin
    screen.rs        // ThemePickerScreen impl Screen (wraps ThemePicker)
    catalog.rs       // moved from crates/savvagent/src/theme.rs (18 entries)
```

Mapping:

- `Plugin::manifest()` declares: Slash(`theme` with `list` arg),
  Screen(`themes.picker`), Theme(18 entries).
- `Plugin::handle_slash("theme", args)`:
  - `args = []` → `Effect::OpenScreen { id: "themes.picker", args: ScreenArgs::ThemePicker { current_slug } }`
  - `args = ["list"]` → `Effect::PushNote { line: format_listing() }` (one styled-line note; preserves v0.8 *intent* of inline listing without overflowing chat history)
  - `args = [slug]` → `Effect::SetActiveTheme { slug, persist: true }`
- `Plugin::create_screen("themes.picker", ScreenArgs::ThemePicker { current_slug })`
  constructs `ThemePickerScreen { inner: ThemePicker::new(current_slug, &CATALOG) }`.
- `ThemePickerScreen::on_key(key)` translates the lifted
  `ThemePicker::on_key` outcome:
  - `Stay` → `vec![]`
  - `PreviewTheme(slug)` → `vec![Effect::SetActiveTheme { slug, persist: false }]` (preview = set-without-persist; revert is another SetActiveTheme)
  - `Apply(slug)` → `vec![Effect::Stack(vec![Effect::SetActiveTheme { slug, persist: true }, Effect::CloseScreen])]`
  - `Cancel` → `vec![Effect::Stack(vec![Effect::SetActiveTheme { slug: pre_open, persist: false }, Effect::CloseScreen])]`
- `ThemePickerScreen::tips()` returns the existing footer hint
  ("↑/↓ navigate · Enter apply · Esc cancel") as `Vec<StyledLine>`.

What carries over unchanged: every v0.8 picker test
(`crates/savvagent/src/app.rs` test module + `theme_command_tests` in
the shared HOME_LOCK group) continues to drive `ThemePicker::on_key`
directly. The plugin shell adds one wrapper-level test
(`themes_plugin_handle_slash_opens_picker`) but doesn't replace the
existing coverage.

## Acceptance criteria for v0.9.0

**Trait surface & crate.**

1. `savvagent-plugin` builds with **zero** dependency on `ratatui`,
   `crossterm`, `tokio` runtime, or `anyhow`. CI grep-check enforces this.
2. `Plugin` and `Screen` traits compile under `#[async_trait]` and follow
   all seven WIT portability rules from §9.

**Runtime.**

3. All TUI key handling and rendering flows through the plugin runtime
   — no remaining `InputMode` flat-field branches outside `effects.rs`.
4. `register_builtins()` produces 17 plugin instances; the registry
   exposes them by id with manifests indexed at startup.
5. Each of the five indexes (slash, slot, hook, keybinding, screen) is
   built once at startup and rebuilt on enable/disable.

**Layout & UX parity.**

6. Home view renders: conversation log, tips line, prompt, segmented
   footer (left/center/right). All existing v0.8 footer information is
   preserved (provider+model, sandbox state, turn state, working dir).
7. Every v0.8 slash command is functional and routed through
   `SlashRouter`: `/theme`, `/theme list`, `/theme <slug>`, `/clear`,
   `/save`, `/model`, `/connect`, `/connect <provider>`, `/resume`,
   plus new in v0.9: `/view`, `/edit`, `/plugins`, `/splash`.
8. Theme picker (`/theme`) opens via the screen stack with the same UX
   as v0.8: filter, live preview, Enter to apply + persist, Esc to
   revert. All v0.8 picker tests pass without modification.
9. Plugin manager screen (`/plugins`) lists all 17 plugins, shows
   kind/version/contributions, allows toggling Optional plugins,
   persists to `~/.savvagent/plugins.toml` atomically.

**Hooks.**

10. `Host::run_turn` emits `TurnStart` / `TurnEnd` / `ToolCallStart` /
    `ToolCallEnd` / `PromptSubmitted` through the dispatcher.
    `internal:splash` switches from poll-loop to `on_event(Connect)`.

## Testing strategy

- **Per-plugin unit tests** in
  `crates/savvagent/src/plugin/builtin/<name>/tests.rs` — drive
  `Plugin::handle_slash` / `Screen::on_key` directly with synthetic
  inputs, assert returned `Vec<Effect>`. No `App` fixture needed.
- **Runtime tests** in `crates/savvagent/src/plugin/tests.rs` —
  `apply_effects` against a fake `App`; effect ordering for `Stack`;
  screen-stack push/pop; slot routing with priority; hook dispatch
  ordering; bounded re-entrancy for `Effect::RunSlash` (depth > 4 is a
  hard error).
- **Index-build tests** — feed two plugins that conflict on a slash
  name and assert startup error. Feed two plugins that conflict on a
  theme slug and assert later-wins with warning.
- **plugins.toml round-trip** — write, read, schema_version mismatch,
  unknown plugin id, missing file, malformed TOML.
- **Carry-over tests** — every existing v0.8 test continues to pass
  without modification (regression guarantee). `app::tests` and
  `theme_command_tests` keep their `HOME_LOCK` discipline.
- **No new integration test infra.** The existing test pattern
  (in-crate unit tests + shared `HOME_LOCK`) absorbs everything.
- **WIT-portability CI grep** — `savvagent-plugin/Cargo.toml` cannot
  introduce `ratatui|crossterm|tokio = ` lines under `[dependencies]`
  (re-checking allowed under `[dev-dependencies]`).

## Risk register

- **PR 3 (screen-stack) is the load-bearing change.** It deletes
  `InputMode` and shifts every modal onto the stack. Mitigation: PR 3
  includes splash + palette as the first two stack consumers, giving
  end-to-end signal that the stack works before PRs 4–6 pile more
  screens on.
- **PR 7 (hook dispatch) crosses into `savvagent-host`.** Mitigation:
  keep the change minimal — host emits `HostEvent`s into a channel the
  TUI drains. No host-side plugin awareness; the host doesn't link
  against `savvagent-plugin` for *trait* calls, only for *type
  definitions*.
- **v0.8 theme picker tests rely on `HOME_LOCK`-serialized HOME
  mutations.** Splitting picker state into `ThemePicker` (PR 5.5)
  preserves this; the plugin shell (PR 6) doesn't introduce new global
  state.
- **`Effect::RunSlash` re-entrancy.** A plugin's slash handler can emit
  another slash. Bound depth at 4 with a hard error on exceed. Logged
  with the chain to make debugging tractable.

## Out of scope (deferred to v1.0+)

- WASM loader. No `wasmtime`/`wasi` integration in v0.9.0.
- `.wit` file. Not shipped in v0.9.0; the trait surface is designed to
  be portable but no port is performed.
- Third-party plugin discovery. `register_builtins()` is the only entry
  point.
- Sidebar UI. Slot abstraction is general (`home.sidebar.*` would Just
  Work) but no built-in plugin contributes to it in v0.9.0.
- Streaming-delta hooks (`on_token`, `on_chunk`, `on_partial_message`).
  Hook frequency capped at once-per-turn-event.
- Hot-reload. Toggling a plugin in the manager applies immediately to
  indexes but does not unload Rust code.
- `PluginEnabled`/`PluginDisabled` hooks. Manager toggles do not emit
  hook events in v0.9.
- Plugin-defined `ScreenLayout` variants. Closed set in v0.9.0.
- Plugin-defined `HookKind` variants. Closed set in v0.9.0.
- Cross-plugin dependencies (manifest `requires: [...]`). Not modeled
  in v0.9.0; built-ins reference each other implicitly via slash names
  and screen IDs.
