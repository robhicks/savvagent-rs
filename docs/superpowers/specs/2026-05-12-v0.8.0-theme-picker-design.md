# Theme picker — v0.8.0 design

Date: 2026-05-12
Status: approved, pending plan + implementation
Roadmap: v0.8.0 (#26)
Supersedes: nothing
Related: PR #33 (ratatui-themes catalog adoption)

## Problem

PR #33 (`feat(tui): adopt ratatui-themes catalog`) introduced 18 selectable
themes (3 hand-rolled built-ins + 15 upstream catalog entries). The user-
facing surface for picking one is the slash command:

```
/theme            # list all themes inline in the conversation
/theme list       # same as above
/theme <slug>     # switch to the named theme
```

This surface has two failure modes in practice:

1. **The inline `/theme list` output overflows the conversation viewport.**
   The conversation log (`render_log` in `ui.rs:461`) renders top-aligned
   without auto-scroll, so an 18-row listing pushed via `push_note` ends
   up below the visible area when there is any prior chat history. The
   user perceives "only built-ins are listed" because the catalog section
   is rendered but off-screen.

2. **`/theme <slug>` via the command palette is a 5-step path with a
   silent failure mode.** The intended workflow is `/` → type `theme` →
   Enter (prefill) → type `light` → Enter (submit). The user's natural
   muscle-memory (`/` → type `theme light` → Enter) instead populates the
   palette's filter — which matches no commands, so Enter silently closes
   the palette and `app.active_theme` is never updated. The user
   correctly observes that "switching themes does not work" because the
   command never executed.

Both failure modes have the same root cause: pushing list-shaped content
through a chat-message channel was the wrong UI affordance. A dedicated
modal picker — the standard pattern already used for command palette,
provider selection, transcript resume, and bash-network confirmation —
solves both.

## Approach

Introduce `InputMode::SelectingTheme`, a new variant of the TUI's modal
input-mode state machine. It mirrors the existing patterns
(`InputMode::CommandPalette`, `InputMode::SelectingProvider`,
`InputMode::SelectingTranscript`, `InputMode::BashNetworkPrompt`):

- One field per piece of picker state on `App` (no separate sub-struct
  abstraction — flat fields match the existing modals).
- One key-event handler arm in `main.rs`.
- One render branch in `ui.rs`.

No new abstractions are introduced. The picker is a concrete instance of
an existing pattern.

### Why not a generic "pick from list" widget?

Considered. The provider picker, transcript picker, and theme picker have
overlapping shapes (centered modal, list, ↑/↓ + Enter + Esc) but diverge
on:

- live preview semantics (theme: yes; provider: no)
- the item type rendered (theme slug; provider display name; transcript
  preview multi-line)
- the bottom-of-modal hint text
- the action of Enter (theme: save + apply; provider: kick off API key
  entry; transcript: resume)

A generic widget would have to model all of these as configuration. The
overhead is not justified for three callers. Each picker stays bespoke;
shared rendering helpers (e.g. `centered_rect`) are already extracted.

## State

Three new fields on `App`:

```rust
/// Cursor row in the filtered theme list. Skips decorative section-
/// header rows; only indexes selectable entries.
pub theme_picker_index: usize,

/// Substring filter on slug, narrowing the displayed catalog.
/// Empty string means "show everything".
pub theme_picker_filter: String,

/// Snapshot of the active theme at picker-open time. Restored on Esc
/// so live preview is fully undoable.
pub theme_picker_pre_theme: crate::theme::Theme,
```

## Flow

### Opening the picker

The slash dispatch routes `/theme` and `/theme list` (and the command
palette's `Execute("/theme")` outcome) to a new helper
`open_theme_picker(app)`:

```text
1. app.theme_picker_pre_theme = app.active_theme
2. app.theme_picker_filter = ""
3. app.theme_picker_index = index_of_active_in_selectable_rows()
4. app.input_mode = InputMode::SelectingTheme
```

Pre-flight: if `app.is_loading` is true, push a note ("can't switch
themes during an in-flight turn — wait for it to finish") and return
without opening, mirroring the `/bash` pre-flight (#22 PR4 pattern).

### In-modal interactions

Each key event in `SelectingTheme` mode:

| Key            | Effect                                                                |
|----------------|-----------------------------------------------------------------------|
| ↑              | `theme_picker_index = max(0, index - 1)`, then live-preview-apply     |
| ↓              | `theme_picker_index = min(filtered.len()-1, index + 1)`, then live    |
| Backspace      | Pop char from `theme_picker_filter`, clamp index, live                |
| printable char | Append to `theme_picker_filter`, clamp index, live                    |
| Enter          | `theme::save(app.active_theme)`, return to `Editing`, push note       |
| Esc            | `app.active_theme = pre_theme`, return to `Editing`, no save/no note  |

"Live-preview-apply" means: after updating the index or filter,
`app.active_theme = filtered_themes[theme_picker_index]`. The next frame
re-renders with the new palette so the chrome behind the modal shifts in
real time.

### Closing

Two paths:

- **Enter (commit)** — Save via `theme::save(app.active_theme)`. On
  success, push `theme set to '<slug>'`. On save failure, push the same
  fall-back note used by the existing handler today (`"theme '<slug>'
  applied for this session, but persistence failed: <e>"`). Return to
  `InputMode::Editing`.

- **Esc (cancel)** — Restore `app.active_theme = pre_theme`. Do NOT save
  (`theme.toml` stays at whatever it was). No conversation note. Return
  to `InputMode::Editing`.

## Render

New branch in `ui::render`, parallel to the existing
`SelectingProvider` / `SelectingTranscript` branches:

```text
┌─ Pick a theme · /<filter> ──────────────┐
│   built-in:                             │
│     dark           (active)             │
│     light                               │
│ >   high-contrast                       │ ← cursor row, live-previewed
│   catalog (ratatui-themes):             │
│     dracula                             │
│     one-dark-pro                        │
│     nord                                │
│     catppuccin-mocha                    │
│     ...                                 │
│ [↑/↓] move  [type to filter]  [Enter] select  [Esc] cancel │
└─────────────────────────────────────────┘
```

- Size: same `centered_rect(60, 50, area)` as the existing pickers (60%
  wide, 50% tall).
- Section headers (`built-in:`, `catalog (ratatui-themes):`) render as
  muted, non-selectable rows. ↑/↓ skip them; `filtered_themes` returns
  only actual `Theme` values, never headers.
- Title bar: ` Pick a theme ` when filter is empty;
  ` Pick a theme · /<filter> ` when non-empty. `(active)` marker on
  whichever row equals `pre_theme`.
- Bottom title: standard `[↑/↓] move  [type to filter]  [Enter] select
  [Esc] cancel` (right-aligned, matching the other pickers).
- All chrome derived from the current `palette` — the modal itself
  participates in the live preview.

### Empty filter result

If the filter matches no themes (e.g. user types `xyz`), the body shows
a single centered muted line: `no themes match '<filter>'`. ↑/↓ and
Enter are no-ops in that state. Esc closes normally (restoring
`pre_theme`).

## Slash-command surface

| Invocation                  | Behavior                                                |
|-----------------------------|---------------------------------------------------------|
| `/theme`                    | Opens picker. No inline output.                         |
| `/theme list`               | Opens picker (same as `/theme`). No inline output.      |
| `/theme <slug>`             | Direct set + save, no picker. Existing behavior.        |
| `/theme <unknown-slug>`     | Push `theme '<slug>' not found — run /theme list...`    |
| Command palette → `/theme`  | `CommandSelection::Execute("/theme")` opens the picker. |

Implementation note: today `/theme` is registered with `needs_arg: true`
in the commands list. That has to flip to `needs_arg: false` so the
command palette executes `/theme` directly on Enter (giving us the
picker) rather than prefilling the prompt. The slug-typing escape hatch
remains available by typing the full command directly in the prompt.

### Removed surface

- The inline `push_note("themes:")` + section-header + per-theme rows in
  `handle_theme_command`.
- The `push_theme_section` helper in `main.rs` (used only by the inline
  list).
- The "not found" message no longer enumerates a (truncated) catalog
  — it already points at `/theme list`, which now opens the picker.

## Edge cases

1. **Filter narrows to one theme, user presses Enter.** Commits that
   one theme. Standard happy path.
2. **Filter narrows to zero themes, user presses Enter.** No-op. Cursor
   stays on the "no themes match" hint. Esc still cancels.
3. **Filter shrinks the list below the current index.** Clamp
   `theme_picker_index` to `filtered.len().saturating_sub(1)`. If the
   list is empty, index is 0 but unused.
4. **Picker invoked when `app.is_loading == true`.** Pre-flight refuses
   with a note. The user can re-invoke after the turn completes.
5. **Picker invoked during a permission prompt / bash-network prompt.**
   Same pre-flight: refuse with a note. The pending modal must be
   resolved first.
6. **`theme::save` errors at Enter time.** Push the existing
   persistence-failed note. The session theme stays as whatever the
   user committed; the next session reads `theme.toml` (which still
   has the old value).
7. **Backspace on empty filter.** No-op.
8. **Active theme is not in the filtered list.** Live preview still
   applies the highlighted theme. On Enter, the new theme persists; on
   Esc, the pre-theme is restored regardless of whether it's currently
   in the filtered list.
9. **Decorative section headers in the filtered view.** Headers render
   only when the corresponding section has at least one matching theme.
   If `theme_picker_filter = "drac"` matches only `dracula`, the
   `built-in:` header is suppressed and only `catalog:` shows.

## Testing

### Unit (in `app.rs` tests)

- `open_theme_picker` snapshots `pre_theme`, zeros filter, sets index
  to active.
- Filter substring narrows the list correctly (e.g. `"drac"` →
  `[Dracula]`).
- Index clamps when filter shrinks the list.
- Cursor movement (`theme_picker_cursor_down`, `_cursor_up`) updates
  `app.active_theme` to the selected filtered theme.
- Commit (`theme_picker_confirm`) does NOT restore `pre_theme`.
- Cancel (`theme_picker_cancel`) DOES restore `pre_theme`.
- Section-header rows are not selectable: cursor down past the last
  built-in moves to the first catalog item, not the header.

### Integration (in `main.rs` tests, alongside the existing `theme_command_tests`)

- `/theme` and `/theme list` enter `SelectingTheme` mode (not push the
  inline list as today).
- `/theme dracula` still switches directly (no picker), and persists.
- After picker commit: `theme.toml` contains the chosen slug.
- After picker cancel: `theme.toml` is unchanged.
- After picker commit with persistence failure: in-session theme is
  the chosen one; the "persistence failed" note is pushed.
- `/theme` invoked while `app.is_loading == true` is refused with a
  note; `input_mode` stays `Editing`.

### Manual verification (documented in PR body, not codified)

Run `cargo run` from the worktree and confirm:

- `/theme` opens the modal.
- ↑/↓ shifts the chrome behind the modal in real time.
- Type `drac` — list narrows to Dracula; press Enter — theme persists
  to `~/.savvagent/theme.toml`.
- Reopen with `/theme` — Dracula is marked `(active)`.
- Type a filter that matches nothing; press Esc — the active theme
  reverts to what it was before opening (no false-positive commit).

## Out of scope

- A keyboard shortcut to open the picker without typing `/theme` (e.g.
  Ctrl-T). Defer to a future UX pass if the slash command proves
  insufficient.
- Cycling through themes with `]`/`[` keys outside the picker. Same
  rationale.
- A theme preview surface separate from the live preview (e.g. a
  swatch grid). Live preview against the actual TUI chrome is more
  informative than an isolated swatch.
- Restoring the "show every theme inline as scrollable chat" output
  for scripting use cases. The slug-typing path (`/theme <slug>`) is
  the supported scripting escape hatch; if a need for enumerating
  themes from outside the TUI appears, a separate command (e.g.
  `savvagent themes`) is the right shape — not an inline chat-log
  dump.
- Per-tool theme overrides, theme cycling animations, theme-aware
  ANSI fallbacks for non-truecolor terminals.
